## Why

The project is pinned to CPython 3.12 (`requires-python = ">=3.12,<3.13"`). Moving to
3.14 gets us newer-runtime improvements (per-interpreter GIL groundwork, faster
interpreter, improved error messages, template strings / PEP 750, deferred annotations
by default) and keeps us on a supported runtime as 3.12 ages toward security-only status.

This is a **runtime bump, not a code migration** — there is almost no 3.12-specific
syntax in `backend/` or `frontend/` to rewrite. The risk is entirely in the
**dependency wheel availability** (especially the index-pinned torch/torchvision GPU
wheels) and in the half-dozen places where `3.12` is hardcoded. Do the pre-flight
wheel check *first*; do not touch any pin until it passes.

## The hard gate: binary wheels for cp314

The blocker is not our code — it is whether every C-extension dependency publishes a
`cp314` wheel for our platforms (Linux x86_64, and the Docker `bookworm-slim` glibc
target). Verify **before** changing any pin:

| Dependency | Source | Risk | How to verify |
|---|---|---|---|
| **torch** `>=2.8,<3.0` | `download.pytorch.org/whl/cu128` **and** `/whl/cpu` (index-pinned, see `pyproject.toml` `[tool.uv.sources]`) | **HIGHEST** — PyTorch ships its own wheels per (python, CUDA) combo on a slow cadence. A cp314 cu128 wheel for torch 2.8 may not exist. | Browse `https://download.pytorch.org/whl/cu128/torch/` and `.../cpu/torch/` for a `cp314` build of a `2.8.x`/`2.9.x` release. **Both** indexes must have it (cpu + gpu variants). |
| **torchvision** `>=0.23,<1.0` | same two indexes | HIGH — must match torch's ABI/python tag exactly (`operator torchvision::nms does not exist` is the mismatch symptom). | Same indexes, matching `cp314` + same torch minor. |
| **pydantic-core** (transitive, just bumped to 2.46.4) | PyPI (maturin/Rust) | Medium — Rust wheels usually land fast. | `pip index versions pydantic-core` / check PyPI files for `cp314`. |
| **numpy**, **sentence-transformers** deps, **weaviate-client** (4.22), **streamlit**, **pymupdf**, **pillow** | PyPI | Low–Medium — popular pkgs usually have cp314 wheels by now, but PDF/image native libs lag. | `uv pip install --dry-run` against a 3.14 venv (see step 0). |

**If torch/torchvision have no cp314 cu128 wheel: stop.** The GPU variant cannot be
built and the upgrade is blocked until PyTorch ships one. Document the finding and
park this plan — do not proceed with a CPU-only downgrade as a workaround without
explicit sign-off (it would silently disable local GPU dev).

## Steps

0. **Pre-flight wheel check (gating).** Create a throwaway 3.14 venv and dry-run the full
   resolve for *both* variants before changing any committed file:
   ```sh
   uv python install 3.14
   uv venv --python 3.14 /tmp/py314 && \
     uv pip install --python /tmp/py314 --dry-run \
       --index https://download.pytorch.org/whl/cu128 \
       'torch>=2.8,<3.0' 'torchvision>=0.23,<1.0'
   # repeat with /whl/cpu for the CPU variant
   ```
   Resolve must succeed for **both** cu128 and cpu. If it fails on torch/torchvision →
   stop, record the blocker, park the plan.

1. **Bump the source-of-truth pins**: `pyproject.toml:7` (`requires-python`),
   `.python-version`, `pyrightconfig.json:17`.

2. **Re-lock and sync both variants** (uv.lock is the committed artifact; CLAUDE.md
   dep flow). The just-added `run_uv.sh` self-update keeps local uv on latest:
   ```sh
   uv lock
   make sync                            # default → PyPI/CUDA torch on 3.14
   rm -rf .venv && make sync SYNC_EXTRA="--extra cpu"   # verify the CPU variant resolves
   rm -rf .venv && make sync            # back to the GPU default
   uv pip check
   ```
   Regenerate the derived requirements scan: `make export-reqs` (gitignored output).

3. **Update the Docker images**: `docker/app.Dockerfile` (builder + runtime + comment).
   Rebuild and smoke-test the app stack — `make stack-up`, hit `http://localhost:8501`,
   run one `make ingest` + `make cli ARGS='--question "..."'` round-trip.

4. **Update CI**: `.github/actions/setup-uv/action.yml` (the shared composite — this is
   where most workflows get Python), plus the two CodeQL workflows that pin inline
   (`codeql.yml`, `codeql_local.yml`). Grep `.github/` once more for stray `3.12`.

5. **Update the devcontainer**: `.devcontainer/Dockerfile` base image (bullseye→bookworm,
   3.14). Rebuild the container; confirm `python -V` and that the editable install works.

6. **Run the full gate**: `make pre-commit`, `make unit`, `make integration` (stack up),
   `make pyright`. Watch for new DeprecationWarnings escalated by the suite's
   `error::DeprecationWarning:backend|frontend` policy — 3.14 deprecates/removes some
   stdlib (e.g. continued `asyncio` child-watcher removals, `pkgutil`/`typing` shims).
   Fix any that fire from our code; suppress third-party ones via the documented
   `filterwarnings` `ignore::...` escape hatch.

## Rollback

Single revert of the bump commit restores `requires-python`, `.python-version`,
`pyrightconfig.json`, the Dockerfiles, CI, and devcontainer together. Re-run
`rm -rf .venv && make sync` to rebuild the 3.12 venv from the reverted `uv.lock`. Keep the bump as
one focused commit (plus the `uv.lock` change) so rollback is atomic.

## Out of scope

- No application-code refactors to *adopt* 3.14 features (t-strings, deferred annotations
  as behavior) — this plan only moves the floor. Adopt-new-syntax work is a separate task.
- No free-threaded (`python3.14t`, no-GIL) build — that is a distinct, higher-risk track
  and most of our native deps (torch especially) are not validated free-threaded.
